# 修复 Bug

## 核心原则

```
不追踪根因不许动手，不写测试不许修复，不验证影响不许提交
```

本技能将三个能力组合成一条流水线：
- **GitNexus**：用知识图谱秒级定位调用链、影响面和可疑代码
- **系统化调试**：强制根因调查纪律，禁止猜测式修复
- **TDD + 验证**：修复前必须有失败测试，完成后必须验证影响面

**违反任何阶段的字面要求，就是违反技能的精神。**

## 适用边界

**使用：**
- 测试失败、生产 bug、异常行为、性能问题、构建失败
- 时间紧迫时（越急越容易猜测式修复，越需要走流程）
- "看起来很简单"的 bug（简单 bug 也有根因）

**不适用：**
- 新功能开发（用 `brainstorming` + `writing-plans`）
- 纯重构（用 `gitnexus-refactoring`）
- 配置变更、文案修改等无行为变化的改动

## 流水线

```
阶段 1：侦察（GitNexus 定位）
  ↓
阶段 2：根因（系统化调查）
  ↓
阶段 3：影响面（修复前评估）
  ↓
阶段 4：TDD 修复（红-绿-重构）
  ↓
阶段 5：验证（提交前门控）
```

每个阶段必须完成后才能进入下一个。

---

## 阶段 1：侦察

**目标：** 用 GitNexus 快速建立 bug 周围的代码地图。

```
1. gitnexus_query({query: "<错误信息或症状关键词>"})
   → 找到相关执行流（Processes）和可疑符号（Symbols）

2. gitnexus_context({name: "<可疑符号>"})
   → 查看调用者（谁触发了它）和被调用者（它依赖什么）

3. 如果调用链不清晰 → gitnexus_cypher 自定义追踪
```

**通过标准：** 能说出"bug 在哪个符号、属于哪个执行流、上下游是谁"。

> 如果 GitNexus 提示 "Index is stale" → 先跑 `npx gitnexus analyze`。

---

## 阶段 2：根因调查

**铁律：在提出修复方案之前，务必先找到根本原因。**

### 2.1 阅读错误

- 完整阅读错误信息和堆栈跟踪，不要跳过
- 记下行号、文件路径、错误码
- 错误信息往往直接包含解决方案

### 2.2 稳定复现

- 能可靠触发吗？具体步骤是什么？
- 无法复现 → 收集更多数据，不要猜

### 2.3 用 GitNexus 追踪数据流

```
gitnexus_context({name: "<出错的符号>"})
→ Incoming calls: 谁传入了错误数据？
→ Outgoing calls: 它依赖的外部调用是否异常？
→ Processes: 它在哪个执行流的第几步？
```

沿调用链反向追踪：
1. 错误值从哪里产生的？
2. 谁用错误值调用了这里？
3. 持续向上追踪直到找到源头
4. **在源头修复，而不是在症状处修复**

### 2.4 检查近期变更

- `git diff`、最近的提交、新依赖、配置变更
- `gitnexus_detect_changes()` 可查看当前改动影响了哪些符号和流程

### 通过标准

能清晰回答："什么出了问题，为什么出问题，源头在哪里。"

---

## 阶段 3：影响面评估

**修复前必须知道爆炸半径。**

```
gitnexus_impact({target: "<要修改的符号>", direction: "upstream"})
→ d=1（直接破坏）：必须更新
→ d=2（可能影响）：需要测试
→ d=3（传递效应）：需要关注
```

| 影响范围 | 风险等级 | 操作 |
|---------|---------|------|
| <5 符号，少量流程 | LOW | 直接修复 |
| 5-15 符号，2-5 流程 | MEDIUM | 修复后跑相关流程的测试 |
| >15 符号或大量流程 | HIGH | 修复后跑全量测试，考虑分步修复 |
| 关键路径（认证、支付） | CRITICAL | 和用户讨论方案后再动手 |

**通过标准：** 列出所有 d=1 的依赖方，明确风险等级。

---

## 阶段 4：TDD 修复

### 4.1 红灯 — 写失败测试

写一个最简化的测试复现 bug：

```typescript
// 示例：空邮箱被接受
test('rejects empty email', async () => {
  const result = await submitForm({ email: '' });
  expect(result.error).toBe('Email required');
});
```

**运行测试，确认它失败。** 失败原因必须是功能缺失，不是拼写错误。

> **测试反模式警告：** 写测试时注意不要测试 mock 行为、不要在生产类中添加仅测试用方法、不要在不理解依赖的情况下使用 mock。详见 `superpowers:test-driven-development` 中的 testing-anti-patterns。

### 4.2 绿灯 — 最少代码修复

写最简单的代码让测试通过。不顺便优化，不捆绑重构。

### 4.3 验证绿灯

```bash
npm test path/to/test.test.ts
```

- 新测试通过
- 其他测试没有被破坏

### 4.4 重构（如需要）

消除重复、改善命名。保持测试绿灯，不添加新行为。

### 失败 3 次以上？

停下来质疑架构。每次修复都暴露新问题 = 架构有误。和用户讨论，不要继续盲修。

---

## 阶段 5：提交前验证

**门控：没有证据，不许宣称完成。**

```
1. 运行相关测试套件 → 看到：0 failures
2. gitnexus_detect_changes({scope: "staged"})
   → 确认变更范围与预期一致
   → 确认没有意外影响其他流程
3. 构建（如果适用） → exit 0
```

| 结论 | 需要 | 不够格 |
|------|------|--------|
| Bug 已修复 | 测试输出：0 failures | "改了应该好了" |
| 没有回归 | 完整测试套件通过 | 只跑了改动的测试 |
| 影响可控 | detect_changes 确认范围 | "应该没影响别的" |

**红线信号（出现就停下来重做）：**
- 使用"应该"、"大概"、"似乎"
- 没跑验证就说"搞定了"
- 信任之前的运行结果而非重新运行
- 想着"就这一次跳过"

---

## 红线 — 停下来

如果你发现自己在想：
- "先临时修一下，以后再排查"
- "试着改改 X 看看行不行"
- "一次改多个地方跑测试看看"
- "大概是 X 的问题，让我修一下"
- "我不完全理解，但这应该能行"
- "再试一次"（已经尝试了 2 次以上）
- 没有用 GitNexus 追踪就动手修复
- 没有写失败测试就开始修复
- 没有跑影响面分析就提交

**以上都意味着：停下来。回到对应阶段的起点。**

---

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要走流程" | 简单 bug 走流程更快。阶段 1-3 只需几分钟。 |
| "紧急情况没时间" | 系统化修复比反复猜测更快。 |
| "GitNexus 索引可能不准" | 不准就重新 analyze，不是跳过。 |
| "先修再补测试" | 后补测试立即通过，什么也证明不了。 |
| "影响面分析太麻烦" | 不分析就提交，出了问题更麻烦。 |
| "我看出问题了" | 看到症状 ≠ 理解根因。追踪数据流。 |
| "这次情况特殊" | 没有例外。 |

## 技能集成

### 前置技能

- `superpowers:systematic-debugging` — 根因调查的完整方法论；本技能将其与 GitNexus 工具整合
- `superpowers:test-driven-development` — TDD 红-绿-重构纪律；本技能在阶段 4 直接调用

### GitNexus 技能映射

各阶段对应 GitNexus 子技能的详细工作流：

| 阶段 | GitNexus 技能 | 用途 |
|------|-------------|------|
| 阶段 1 侦察 | `gitnexus-exploring` | 探索代码结构、查找执行流 |
| 阶段 1-2 侦察/根因 | `gitnexus-debugging` | 调用链追踪、根因定位 |
| 阶段 3 影响面 | `gitnexus-impact-analysis` | 爆炸半径评估、风险分级 |
| 阶段 4 重构 | `gitnexus-refactoring` | 安全的多文件协调重构 |
| 阶段 5 验证 | `gitnexus-pr-review` | 提交前的变更影响复查 |
| 全程 | `gitnexus-guide` | 工具参考、Schema 查询 |
| 全程 | `gitnexus-cli` | 索引管理（analyze/status/clean） |

### 后置技能

- `superpowers:verification-before-completion` — 提交前的最终门控；本技能在阶段 5 内联执行
- `gitnexus-refactoring` — 如果修复涉及重构，在阶段 4 重构步骤中调用

### 交叉技能

- `page-development-workflow` — 如果 bug 出现在页面开发流程中，阶段 4 的 TDD 修复同样遵循该工作流的 TDD 纪律

### GitNexus 工具速查

完整的 GitNexus 工具参数和示例见 `references/gitnexus-toolkit.md`。
