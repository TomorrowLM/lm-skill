# Writing Skills

## 核心判断

把技能当作给 AI 的可执行工作流文档，而不是给人的说明书。

编写技能就是把 TDD 应用于文档：先观察失败，再写最小规则，再堵住漏洞。

**铁律：没有失败的测试，就不要写技能。**

### 什么时候用

- 你要创建新技能或重写现有技能
- 你要教 AI 一个它默认不会稳定做到的技术、模式或流程
- 你担心 AI 会在压力下绕过规则或合理化违规
- 你需要验证技能在真实任务里是否真的会被正确使用

### 不适用场景

- 一次性解决方案
- 项目局部约定，这类内容放进 CLAUDE.md 或同类项目文档
- 可以直接靠脚本、正则、校验器强制执行的机械约束
- AI 已经充分掌握的通用常识

## 核心原则

- **TDD for docs**：先跑基线失败，再写技能，再复测。
- **上下文窗口是公共资源**：每一段说明都在和系统提示、会话历史、其他技能竞争 token。
- **默认 AI 已经够聪明**：只补充它不知道、且会影响行为的内容。
- **边界比愿景更重要**：与其写"要做什么"，更要写"不要怎么绕过去"。
- **主文件负责控制逻辑**：SKILL.md 保留触发条件、流程、红线和引用条件；展开说明下沉到 references。
- **技能不是叙事文档**：不要写版本故事、设计回顾、背景抒情或面向人的辅助文档。

## 先决定技能放什么

### 技能是什么

技能是可复用的技术、模式、工具或参考指南。

技能不是某次会话的案例复盘，也不是"我上次怎么做成了"的叙事记录。

### 技能类型

| 类型 | 典型内容 | 测试重点 |
|------|----------|----------|
| 纪律执行类 | TDD、完成前验证、编码前设计 | 压力下是否守规矩 |
| 技术类 | 调试方法、等待策略、防御式编程 | 能否在新场景正确应用 |
| 模式类 | 心智模型、识别框架 | 何时用、何时不用 |
| 参考类 | API 文档、命令参考、库说明 | 能否正确检索并应用 |

## 红-绿-重构

> 不要凭空想象 AI 会怎么犯错，先让它犯错，再针对性写规则，最后把绕过去的路径全部堵死。

### 红：先观察失败

先在**没有技能**的情况下运行压力场景，逐字记录 AI 的选择、借口和违规点。没有基线失败就不写技能。

- [ ] 创建压力场景
- [ ] 运行无技能基线并逐字记录失败
- [ ] 提炼合理化借口模式

### 绿：写最小技能

只针对观察到的失败编写规则——解决真实漏洞，不预防假想问题，用最少文字覆盖关键行为分岔。

- [ ] 写最小技能，只修复已观察到的失败
- [ ] 校验 `name` 和 `description` 是否合规
- [ ] 确认主文件只保留控制逻辑
- [ ] 选一个最强示例，而不是堆多个弱示例
- [ ] 用同一场景复测，确认行为已改变

### 重构：堵住漏洞

如果 AI 又找到新借口，补进借口表 + 红线，反复测试到无法绕过。

- [ ] 把新借口补进借口表和红线
- [ ] 检查是否有可下沉到 references 的展开内容
- [ ] 反复测试直到无法再轻易绕过

## 主文件与目录规则

### 目录骨架

```text
skills/
  skill-name/
    SKILL.md
    references/
      templates/
      examples/
      prompts/
      guides/
    scripts/
    assets/
```

### SKILL.md 必须保留

- 触发条件和不适用边界
- 核心原则
- 阶段顺序和关键决策点
- 红线与禁止项
- 什么时候读取哪个 references 文件

### 默认下沉到 references

- 长示例和完整工作流
- 背景解释、成本分析、扩展案例
- 模板、术语表、操作手册
- 不影响执行顺序的补充资料

### 目录红线

- `SKILL.md` 超过 250 行就应主动拆分，不要继续堆
- `references/` 目录本身只放子目录，不直接放散落文件
- 不要创建 README、CHANGELOG、INSTALLATION、QUICK_REFERENCE 这类面向人的辅助文档
- 不要让信息在 SKILL.md 和 references 中重复出现

## 编写规则

### Frontmatter

- 必需字段只有 `name` 和 `description`
- `name` 只使用字母、数字、连字符
- `description` 使用中文描述，直接写清"当需要……时使用"，只描述触发条件和症状
- 不要在 `description` 里总结完整流程
- 所有"何时使用"信息放在 frontmatter，不放到 body 里补救

### Body

- 先写控制逻辑，再写细节
- 用祈使语气，减少歧义
- 一个优秀示例胜过多个平庸示例
- 不要写填空模板，要写可直接适配的具体模式
- 如果技能包含"红线"，统一写成三级标题（`###`）
- 如果技能包含"技能集成"，统一先写一个二级标题 `## 技能集成`，再拆成三级标题 `### 前置技能` 和 `### 后置技能`
- 所有 references 都从 SKILL.md 直接引用，禁止 A -> B -> C 深链
- 超过 100 行的 references 文件，顶部加目录帮助预览

### 流程图

只在这些情况使用小型流程图：

- 决策点不明显
- 容易过早退出循环
- 需要判断 A 和 B 何时切换

不要用流程图承载参考资料、代码示例或线性步骤。

### 示例

好的示例应满足：

- 完整可运行或可直接迁移
- 来自真实场景
- 清楚展示模式为何成立
- 使用最相关的一种语言，不做多语言铺陈

## 如何抵抗合理化

纪律类技能必须显式反击常见借口，而不是只陈述理想规则。

### 写法要求

- 不只写"应该做什么"，还要写"不要怎样变通"
- 直接封堵"只是简单改动""我之后再测""我遵守的是精神"之类借口
- 明确写出：**违反字面，就是违反精神**

### 常见借口表

| 借口 | 现实 |
|------|------|
| 技能显然已经很清晰 | 对你清晰，不等于对另一个智能体清晰 |
| 这只是参考资料 | 参考资料也会遗漏，必须测试检索和应用 |
| 之后再测试也一样 | 后补测试无法证明你写的是正确目标 |
| 这是个简单补充 | 简单改动同样可能把规则打穿 |
| 我遵守的是精神 | 违反字面就是违反精神 |

### 红线

出现以下想法时，停下来重做：

- 先写技能，再补测试
- 编辑技能但不重新测试
- 保留未测试改动作"参考"
- 因为批量处理更高效而跳过逐个验证
- 觉得"这次情况特殊"所以规则可以暂时不算

## 测试方法

### 纪律执行类技能

- 学术性问题：验证是否理解规则
- 压力场景：验证在时间、沉没成本、疲惫等压力下是否仍守规
- 组合压力：同时叠加多种诱因，观察是否找借口绕过

### 技术 / 模式 / 参考类技能

- 应用场景：能否在新问题上用对
- 变体场景：能否处理边界条件
- 反例场景：能否识别不该用的时候
- 覆盖测试：常见任务是否都能被命中

## 技能集成

### 前置技能

- `superpowers:test-driven-development`：定义红-绿-重构的基础纪律；本技能只负责把它迁移到文档写作。

### 后置技能

- 按需与调试、计划、验证、中文文档或中文审查类技能组合，但不要把组合关系写成泛泛列表；只保留会影响执行顺序的集成项。
