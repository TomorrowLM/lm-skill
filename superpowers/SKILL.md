# Superpowers

## 目标

这个 skill 是 superpowers 技能包的总入口。

它本身不应该替代所有子技能，而是负责两件事：

1. 先用统一的工程原则约束行为
2. 再根据用户任务，把工作路由到最合适的 `skills/` 子技能

当用户访问 `/superpowers` 时，不要只停留在泛泛的建议层面。应该先判断任务类型，然后明确告诉用户下一步该使用哪个子技能；如果场景明确，直接进入对应子技能。

## 技能包功能总览

superpowers 技能包覆盖的是一整套工程工作流，而不是单点能力。它主要解决以下几类问题：

1. **需求澄清与设计**：先把想法收敛成可执行设计，而不是直接写代码
2. **计划编写与实施**：把规格拆成可执行任务，并按步骤推进
3. **实现协作与并行分工**：用子代理或并行代理提高复杂任务的执行质量
4. **问题排查与根因定位**：遇到 bug、失败测试或异常行为时系统化调试
5. **测试与验证**：用 TDD 和完成前验证约束实现质量
6. **代码审查与反馈处理**：包括请求审查、接收审查和中文团队审查表达
7. **Git 与交付收尾**：隔离工作区、分支收尾、中文团队工作流与提交规范
8. **文档、技能与工作流编排**：写中文文档、编写技能、运行 YAML 多角色工作流、构建 MCP 服务器

## /superpowers 入口规则

当用户访问 `/superpowers` 时，按下面顺序处理：

1. 在任何实质性响应、澄清问题或实现动作之前，先判断当前任务是否应该进入某个子技能
2. 不要把 `/superpowers` 当作最终答案，而要把它当作路由入口
3. 只要某个子技能有明显适用性，就必须路由过去，不要停留在总入口层面
4. 明确告诉用户“这个场景应该用哪个子技能”
5. 如果用户没有显式调用 `/superpowers`，且场景已经足够明确，可以直接切换到对应子技能执行
6. 如果涉及多个阶段，先调用流程类技能，再调用实施类技能

**硬门槛：** 如果用户显式调用了 `/superpowers`，但没有同时指定具体子技能，那么无论模型主观判断多么明确，都必须先进入候选子路由选择；这条规则优先于“场景已经足够明确时可直接切换”。

## 指令优先级

`/superpowers` 入口需要遵循以下优先级：

1. **用户明确指令**：最高优先级。用户明确要求不用某种流程、只做某件事、或限定输出方式时，优先服从用户指令
2. **superpowers 子技能规则**：当任务已经分流到具体子技能时，按对应子技能执行
3. **/superpowers 总入口规则**：负责判断、分流、宣告和组合技能
4. **默认系统行为**：只有在以上都不冲突时才适用

如果用户明确要求与某个子技能的默认做法冲突，例如要求不要走 TDD、不要创建计划、不要进入 worktree，应优先服从用户指令，但仍要说明本来对应的推荐技能是什么。

## 入口执行流程

当用户访问 `/superpowers` 时，按这个顺序执行：

1. 判断用户任务是否落入某个子技能
2. 如果存在适用技能，优先分流，不要先给泛泛建议
3. 用一句话明确宣告将使用哪个技能处理当前任务
4. 在进入任何实质性分析、提问、计划、实现、调试或审查之前，必须先直接读取对应子技能文件。逻辑目标路径固定为：`skills/superpowers/skills/<skill-name>/SKILL.md`
5. 如果该技能自带检查清单、阶段门控或必须的前置流程，后续严格遵循
6. 只有在确定没有明显适用技能时，才停留在 `/superpowers` 层面做总览式答复

### 子技能文件定位规则

子技能文件读取的完整规则已拆分到参考文件：

- `references/skill-file-resolution.md`

主技能只保留两条硬规则：

1. 已确定子技能后，必须直接读取目标 `SKILL.md`，不要先搜索
2. 如果读取失败，必须明确说明失败并进入最接近的兜底流程

### 选定子技能后的强制动作

一旦已经确定要进入某个子技能，无论是模型自行判断，还是用户通过候选路由明确选择，都必须执行以下动作：

1. 先宣告将使用哪个子技能
2. 立即读取对应子技能文件；若工具要求绝对路径，则用当前工作区根目录拼接 `skills/superpowers/skills/<skill-name>/SKILL.md` 后直接读取
3. 以该子技能规则作为后续动作的直接依据，而不是只沿用 `/superpowers` 总入口的泛化理解
4. 只有在读取完成后，才能继续提问、写计划、实现、调试或审查

如果对应的 `SKILL.md` 缺失、路径异常或不可读取，必须明确告知用户“未能读取该子技能规则”，并说明将采用的最接近兜底流程；不要假装已经按该技能执行。不要把“绝对路径搜索不到”或“相对路径未直接命中”表述成“子技能不存在”；应先检查是否只是工具路径格式不匹配，再说明直接读取失败并进入兜底。

推荐的宣告格式：

> 我将使用 `<skill-name>` 处理这个任务，因为当前场景属于 `<scene>`。

## 候选子路由推断

当用户的输入可以匹配多个子技能，或者只表达了目标、没有明确说明流程时，`/superpowers` 入口不应直接替用户拍板，而应先推断候选子路由，再让用户确认。

如果用户**显式输入了 `/superpowers`**，但**没有同时指定具体子技能**，默认也应优先进入候选子路由选择。此时即使模型主观认为某个子技能“看起来最合适”，也不应直接跳过确认环节。

这里的“没有同时指定具体子技能”包括这类输入：

- `/superpowers` 加一个实现请求
- `/superpowers` 加一个单点修改请求
- `/superpowers` 加一句看起来已经很明确的需求说明

只要用户没有明确写出 `brainstorming`、`writing-plans`、`systematic-debugging`、`subagent-driven-development` 等具体 skill 名，就仍然属于“先候选确认”的场景。

典型场景包括：

1. 用户显式调用 `/superpowers`，但没有写明 `brainstorming`、`writing-plans`、`systematic-debugging` 等具体子技能
1. 用户只说要“生成”“实现”“做一个”某个功能，例如“/superpowers 生成一个轮播图”
2. 用户的需求既可能是设计问题，也可能是实现问题
3. 用户的描述太短，无法确定是要 brainstorm、写计划还是直接实现
4. 输入同时命中多个技能的关键词和场景

在这种情况下，按以下顺序处理：

1. 先根据用户意图推断 2 到 4 个最可能的子路由
2. 对每个候选路由给出一句简短理由
3. 增加一个“自定义路由/都不匹配”的选项
4. 用选项形式让用户明确选择；如果运行环境支持选项式提问工具，必须优先使用工具，而不是直接替用户决定
5. 用户确认后，再宣告将使用哪个技能
6. 立即直接读取对应子技能文件；如果工具要求绝对路径，则用当前工作区根目录拼接 `skills/superpowers/skills/<skill-name>/SKILL.md` 后读取
7. 读取完成后，再进入执行

候选路由的详细规则、分类视图、选项模板、示例和自定义路由处理，统一从 `index.md` 获取。

当需要做多候选子路由推断时，主技能只负责：

1. 判断是否进入候选路由选择
2. 告知用户当前不是唯一匹配场景
3. 按 `index.md` 中的技能分类和路由判断规则生成候选项
4. 等用户选择后再进入对应子技能

特别地，以下情况**不要**直接进入 `subagent-driven-development`、`executing-plans` 或其他实施类技能：

1. 用户只是写了 `/superpowers` 加一个实现诉求，但没有明确说“直接实现”或指定子技能
2. 模型只是基于经验判断“这看起来是单点修改”
3. 模型尚未给用户展示候选子路由并获得确认
4. 模型只是因为截图、现有文件名或局部需求描述看起来很具体，就推断“可以直接实现”

## 入口红线

使用 `/superpowers` 作为入口时，以下行为属于错误：

1. 明明已经存在明显适用的子技能，却还停留在总入口层面泛泛回答
2. 在没有完成技能判断之前，就开始问澄清问题或直接实现
3. 明明是流程问题，却先进入实施类技能
4. 明明是中文团队场景，却没有叠加中文系列技能
5. 明明已经知道该用哪个技能，却不向用户明确宣告
6. 只因为任务“看起来简单”，就跳过技能分流
7. 用户显式调用 `/superpowers` 但未指定子技能时，模型直接跳过候选路由确认
8. 用户已经选定某个子技能，模型却没有读取对应的 `skills/superpowers/skills/<skill-name>/SKILL.md`，或没有把它正确拼成当前工作区下的绝对路径
9. 已经宣告使用某个子技能，却仍按总入口经验直接行动，而不是按该子技能规则执行
10. 用户已经指定子技能，模型却仍先去搜索绝对路径或全局 `SKILL.md` 列表，而不是直接读取目标文件
11. 工具明明要求绝对路径，模型却把相对路径读取失败误判成“文件不存在”

## 默认分流策略

默认分流策略和入口回复模板统一从 `index.md` 获取。

主技能只保留默认动作要求：

1. 判断完成后，不要只返回技能名
2. 先宣告将使用哪个技能
3. 再读取该技能的 `SKILL.md`
4. 然后按该技能规则进入下一步动作

如果用户的场景明确，不要只给列表，直接切换到相应技能继续执行。

## 核心原则

使用这个总入口 skill 时，始终遵循以下原则：

1. 先找事实，再分流
2. 先选对流程，再做实现
3. 先读取已选子技能的规则，再进入该技能的细节动作
4. 不把 `/superpowers` 当成泛化回答模板
5. 能明确路由时，就不要停留在抽象建议层

## 常见反模式

使用这个总入口 skill 时，应避免以下反模式：

1. 用户已经是调试场景，却还停留在泛泛分析，不进入 `systematic-debugging`
2. 用户已经明确要写计划，却还在泛泛讨论，不进入 `writing-plans`
3. 用户要实现，却跳过 `brainstorming` 或 `writing-plans` 直接动手
4. 用户要中文团队输出，却没叠加中文系列技能
5. 只告诉用户“可以用 superpowers”，却不指出具体子技能
6. 用户已经选择了某个子技能，但模型没有读取该技能文件就继续执行

## 产出标准

一个高质量的 `/superpowers` 入口响应通常应满足：

1. 能概括整个技能包是做什么的
2. 能判断用户当前属于哪个场景
3. 能明确指出应该使用哪个子技能
4. 一旦子技能确定，能先读取并遵循对应 `SKILL.md`
5. 必要时说明几个技能如何串联使用
6. 如果场景足够明确，直接进入对应技能，而不是停留在介绍层
