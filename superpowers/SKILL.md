# Superpowers

## 子技能路由表

| 用户场景 | 首选技能 | 何时直接进入 | 常见叠加 |
|---|---|---|---|
| 需求不清、方案未定、边界模糊 | `brainstorming` | 用户在比较方案、收敛范围、讨论取舍时 | 无 |
| 需求明确，但任务较多，需要先拆计划 | `writing-plans` | 用户已知目标，但还没有任务清单时 | `code-rule`（后续会写前端代码时） |
| 已有计划，准备拆任务协作执行 | `subagent-driven-development` | 用户明确表示已有计划，并希望按任务分派执行时 | `code-rule` |
| 已有计划，要求当前会话顺序执行 | `executing-plans` | 用户明确要求留在当前会话逐项推进时 | `code-rule` |
| bug、失败测试、构建异常、行为不明 | `systematic-debugging` | 问题核心是定位原因而不是直接实现时 | 后续可转 `test-driven-development` |
| 要按测试先行实现或修复 | `test-driven-development` | 用户明确要求 TDD，或修复前先补失败测试时 | 无 |
| 准备宣称完成、修复完成、测试通过 | `verification-before-completion` | 已有改动，需要先拿到新鲜验证证据时 | 无 |
| 准备发起正式代码审查 | `requesting-code-review` | 功能已完成，准备 review 时 | `chinese-code-review` |
| 正在处理 review 反馈 | `receiving-code-review` | 用户带着审查意见回来时 | `chinese-code-review` |
| 需要隔离当前开发环境 | `git-worktrees` | 开始开发前，用户要求隔离工作区时 | `git-workflow` |
| 准备合并、提 PR、收尾分支 | `git-finishing-development-branch` | 已验证完成，进入交付阶段时 | `git-commit-conventions`、`git-workflow` |
| 编写中文 README 或技术说明 | `chinese-documentation` | 任务本身是中文文档产出时 | 无 |
| 构建 MCP 服务、工具或集成 | `mcp-builder` | 任务核心是 MCP 设计与实现时 | 可前接 `brainstorming`、`writing-plans` |
| 运行 YAML 多角色工作流 | `workflow-runner` | 已有 workflow 文件，需要直接执行时 | 后接 `verification-before-completion` |
| 新建、修改、压缩、验证 skill 本身 | `writing-skills` | 任务对象就是 skill 文档时 | 无 |
| 涉及代码关系查询、影响分析、调用链追踪、PR 影响评估、安全重构 | `gitnexus-guide` | 用户提到爆炸半径/影响范围/调用链/知识图谱/gitnexus 时，先加载 gitnexus-guide 再由其路由到具体子技能 | brainstorming / writing-plans / systematic-debugging / requesting-code-review |

## 强制前置检查清单

**每次进入本技能后，在执行任何实质性操作（包括提问、搜索、读文件、写代码）之前，必须先完成以下检查：**

```text
□ 步骤1：用户是否通过任何方式调用了本技能但未指定子技能？
   ├─ 是 → 必须执行"候选确认"流程（见下方"候选确认输出要求"），禁止自行推导子技能
   └─ 否 → 跳过（用户已明确指定了子技能名）

□ 步骤2：用户是否明确要求"启动/运行/起服务/跑 workflow"？
   ├─ 是 → 判断是否真是用户显式目标，不是则禁止执行
   └─ 否 → 默认禁止启动任何服务

□ 步骤3：当前响应是否已经产生了实质性输出（候选路由之外的代码/方案/搜索）？
   ├─ 是 → 违反规则，立即停止，改输出候选路由
   └─ 否 → 继续
```

**这三步是强制性的。无论任务看起来多简单、多明确，只要用户没有指定子技能名，步骤1的答案就是"是"，必须输出候选路由让用户选择。不得以任何理由跳过。**

## 默认分流策略

1. 先判断任务是在选流程，还是在做实现；流程不明时，不要直接进入实现类技能。
2. 如果用户通过任何方式调用本技能（包括但不限于 `/superpowers`、`@skills/superpowers/SKILL.md`、文件引用、命令别名），但没有点名具体子技能，必须先给出 2 到 4 个候选子路由，再让用户确认。
3. **步骤2（候选确认）完成之前，禁止输出任何实质性内容**（包括提出澄清问题、分析代码、写计划、搜索方案、展示代码实现等）。候选路由是唯一允许的输出。
4. 当快速确认"需求明确"时，这个判断本身就是一条候选路由，而不是跳过分流的理由。必须仍然输出候选列表让用户确认。
5. 默认不允许直接切到子技能；只有当用户明确指定了子技能名称或显式授权时，才可直接进入目标子技能。
6. 一旦确定子技能，先宣告将使用哪个技能，再直接读取 `skills/superpowers/skills/<skill-name>/SKILL.md`；不要先搜索。
7. 默认禁止因为"顺手验证""看起来应该要跑一下"而启动项目、开发服务器、后台服务或其他长驻命令；只有用户明确要求、任务本身就是运行工作流/服务，或当前任务必须用该命令验证结论时，才允许执行。
8. 用户提到"爆炸半径""影响分析""调用链""知识图谱""gitnexus"或涉及代码关系查询时，先加载 gitnexus-guide，由 guide 内部路由决定是否调用具体 GitNexus 子技能。GitNexus 是条件性增强，非强制；加载前需确认索引有效。
9. 即便允许执行命令，也优先选择最小、最短、可证明当前结论的命令；不要把"启动整个项目"当作默认验证方式。
10. `brainstorming` 默认走终端文本讨论，不默认启用 visual companion 或本地服务；`workflow-runner` 只有在用户明确要求运行工作流时才能真正执行；`git-worktrees` 不得因为进入该技能就自动安装依赖或跑整套基线，除非用户同意或当前任务必须这样做。
11. 如果任务跨多个阶段，先走流程类技能，再进入实施类技能，最后叠加中文规范、审查规范或前端规范。
12. 如果用户明确要求不要走某个默认流程，服从用户指令，但仍要说明原本推荐的技能是什么。
13. 如果技能文件读取失败，必须明确说明失败，并进入最接近的兜底流程；不要假装已经按该技能执行。

## 识别说明

### 如何判断是否唯一匹配

- 用户在问"怎么做""选哪个方案""边界是什么"，优先识别为 `brainstorming`。
- 用户已经给出明确目标，但还没有步骤、文件清单、验证方案，优先识别为 `writing-plans`。
- 用户已经有计划，并强调"开始实现""按计划做""拆任务做"，再在 `subagent-driven-development` 和 `executing-plans` 之间判断。
- 用户描述的是异常、失败、报错、回归、构建不过，优先识别为 `systematic-debugging`。
- 用户说"帮我 review""看这次改动有没有问题"，优先识别为 `requesting-code-review`。
- 用户说"这个 skill 改一下""重写 skill 规则""优化 skill 结构"，优先识别为 `writing-skills`。

**关于"是否唯一匹配"的关键警告：**
- "任务很明确"不等于"可以跳过候选路由"。明确的任务恰好说明可以给出一条清晰的候选路由（如"任务明确 → writing-plans 或 executing-plans"），但**仍然需要用户确认**。
- 唯一允许跳过候选确认的情况是：用户直接说了子技能名称（如"用 executing-plans"）。

### 识别命令执行风险

- 只要任务里出现"启动""运行项目""起服务""开 dev server""跑 workflow""先把项目跑起来看看"，就先判断这是不是用户的显式目标；不是的话，默认不要执行。
- `brainstorming` 的视觉辅助模式、`workflow-runner`、`git-worktrees` 是最容易触发启动或批量命令的入口；进入这些技能前，先确认是否真的需要执行。
- `verification-before-completion` 和 `systematic-debugging` 可以运行验证命令，但它们证明的是结论，不等于可以默认启动整个项目。
- 如果一个结论可以通过单测、构建、静态检查或局部命令证明，就不要升级为启动完整应用。
- 对会长期占用终端的命令，默认视为高成本动作；除非用户明确要求，或任务目标本身就是启动并观察运行态，否则先避免。

### 何时必须候选确认

- 用户通过任何方式调用本技能（命令、文件引用、@路径等），但没有写明具体子技能。
- 一个请求同时像设计问题和实现问题。
- 用户只给目标，没有说明是要讨论、写计划还是直接做。
- 你能想到两个以上合理子路由，而且无法用一句本地判断排除其余路径。

### 候选确认输出要求

1. 给出 4 到 6 个候选子路由。
2. 每个候选只给一句简短理由。
3. 必须包含"自定义路由 / 都不匹配"。
4. 用户确认后，再宣告目标技能并读取对应 `SKILL.md`。

## 红线

### 绝对禁止
默认不允许直接走子技能，除非用户明确指定子技能。

1. 还没完成技能判断，就先提澄清问题、写计划或直接实现。
2. 用户通过任何方式调用本技能但未指定子技能时，跳过候选路由确认。
3. 已经知道该用哪个技能，却不明确宣告。
4. 已选定子技能，却没有先读取对应的 `skills/superpowers/skills/<skill-name>/SKILL.md`。
5. 已宣告某个子技能，却继续按总入口经验行动，而不是按该子技能规则执行。
6. 明明是流程问题，却直接进入 `subagent-driven-development`、`executing-plans` 或其他实施类技能。
7. 把"任务看起来简单"当成跳过分流的理由。
8. 没有用户明确要求时，不允许自行启动项目、开发服务器、后台服务或整套 workflow。
9. 因为进入某个子技能，就自动运行依赖安装、基线测试、项目启动或长时间命令。
10. 工具要求绝对路径时，把读取失败误判成"子技能不存在"。

### 借口与现实

| 借口 | 现实 |
|---|---|
| 这个需求已经很明确，可以直接做 | 只要用户调用本技能且未指定子技能，就必须先候选确认。"需求明确"不是跳过候选路由的理由，它是一个候选路由（建议进入 writing-plans/executing-plans），但仍需用户确认。 |
| 我先问两个问题再分流更高效 | 错。分流发生在任何实质性提问之前。 |
| 我记得这个子技能怎么用，不用再读 | 错。确定技能后必须读取当前版本。 |
| 这是个小改动，不值得走流程 | 小改动也可能路由错误；先分流，再行动。 |
| 我先搜一下路径更稳妥 | 错。已确定技能后，应直接读取目标 `SKILL.md`，不要先搜索。 |
| 我先把项目跑起来看看，通常更快 | 错。默认先选最小验证；启动项目不是默认动作。 |
| 这个技能里本来就有启动命令，所以我可以直接跑 | 错。子技能中的命令示例不等于当前任务已获授权执行。 |
| 先把服务挂起来，后面可能用得到 | 错。长驻命令需要明确目标或明确许可，不能预启动。 |

### 实际违规案例

以下是从真实交互中总结的违规模式，每次进入本技能前应重温：

| 场景 | 行为 | 结果 |
|---|---|---|
| 用户调用 `/superpowers` 说"接入8个接口到dashboard" | 直接选 `executing-plans` 开始实施，跳过候选路由 | ❌ 违反规则第2条：未指定子技能却未给出候选确认 |
| 用户调用 `/superpowers` 说"tab切换接口需要重新请求" | 直接开始修改代码，跳过候选路由 | ❌ 同上 |
| 用户说"筛选条件要注入到列表请求中" | 直接进入实施模式 | ❌ 同上 |

**所有违规的共同特征：**
- 用户通过 `/superpowers` 或技能引用调用本技能
- 用户没有指定具体子技能名
- AI 认为"需求清晰明确"就自动选择了子技能并开始实施
- 没有输出候选路由让用户确认

**纠正措施：只要模式匹配上述特征，立即停止当前输出，改输出候选路由。**
