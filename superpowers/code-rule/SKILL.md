# 前端编码规范

## 核心原则

- **先约束再实现**：在写第一行代码前就要明确规范边界，不要写完后补规范
- **自动化优先于文档**：没有 lint / format / CI 强制执行，规范只是建议
- **不只盯格式**：还要覆盖结构、复用边界、耦合和性能
- **遗留项目渐进收敛**：先硬规则后软规则、先核心模块后边缘模块

## 进入时先判断

本技能有四种使用模式，进入后先确定当前是哪种：

| 模式 | 判断条件 | 做什么 |
|------|---------|--------|
| 实现前约束 | AI 马上要写代码 | 只加载与当前任务直接相关的规则段，约束实现 |
| 制定/修订规范 | 团队要从零建立或升级规范 | 先硬规则再软规则，同步配置自动化 |
| 审查合规 | 已有代码，需要判断是否符合规范 | 按分类逐项检查，标记 `[必须修复]`/`[建议修改]`/`[仅供参考]` |
| 遗留治理 | 老项目要逐步收敛 | 先核心模块，禁止一次性全量推开 |

## 执行流程

### 实现前约束（高频路径）

1. 诊断当前技术栈（Vue/React）、已有规范文档、lint/format 配置
2. 读取 `references/coding-standards.md` 中与当前任务直接相关的部分（不要全量加载）
3. 如涉及组件拆分、耦合治理或性能策略，再读取 `references/optimization-dimensions.md`
4. 写代码过程中持续自查，不允许等写完再返工

### 制定/修订规范

1. 先定硬规则：命名、目录结构、组件边界、自动化要求
2. 再定软规则：注释风格、可读性、推荐模式
3. 配置自动化：ESLint + Prettier + StyleLint + Husky/lint-staged + commitlint
4. 同步更新规范文档和自动化规则，不允许脱节

### 审查合规

1. 按分类检查：命名 → 结构 → 样式 → 逻辑 → 复用 → 性能
2. CI 至少执行 lint、typecheck、测试
3. 审查结论必须标注优先级，不要只写"LGTM"

### 持续优化

- 周期性审查高内聚低耦合、工具链统一性
- 根据实际性能指标决定懒加载、拆包、CDN 等策略
- 框架或技术栈升级后同步更新规范，不允许长期放任过期

## References 读取条件

| 文件 | 什么时候读 |
|------|-----------|
| `references/coding-standards.md` | 需要查看 Vue/HTML/CSS/JS/注释/目录结构的编码细则时。默认以阿里巴巴前端规范为基础 |
| `references/optimization-dimensions.md` | 需要评估高内聚低耦合、工具链统一、组件库建设或性能优化策略时 |

## 技能集成

### 前置技能

- `writing-plans` / `executing-plans` / `subagent-driven-development` / `test-driven-development`：在开始实现前先进入本技能约束写法

### 后置技能

- `verification-before-completion`：在改造完成后执行收尾验证
- `git-commit-conventions`：规范提交信息和 git 流程

## 红线

- 不要等代码写完后才套规范
- 不要写无注释函数，尤其是业务函数、工具函数和组件方法
- 不要只有规范文档，没有 lint / format / CI 强制执行
- 不要在遗留项目里一次性强推全部规则
- 不要把规范工作缩减成纯格式化问题
- 不要忽略组件封装、复用和架构边界
- 不要在框架或技术栈升级后长期放任规范过期

**违反字面规则，就是违反规则精神。**
