# 前端业务技术开发文档

## 核心判断

这个技能用于产出“前端怎么做”的业务技术文档，不是泛化的中文排版规范，也不是代码实现计划。

文档必须服务于三个目标：

1. 让评审者快速判断方案是否合理。
2. 让开发者知道页面、组件、接口、状态和风险怎么落地。
3. 让后续实现与联调可以对照文档追踪。

## 何时使用

- 用户要写前端技术方案、开发文档、技术评审文档。
- 用户给了业务需求、页面说明、原型、接口信息，希望整理成前端实现文档。
- 用户已经知道要做什么，但需要把页面拆分、组件职责、接口和风险写清楚。
- 用户提供了模板，希望你按模板生成适合评审或开发的文档。

## 不适用场景

- 只是在写 README、公告、操作说明或通用中文文档。
- 只是需要任务清单或执行计划，而不是技术方案文档。
- 只是要代码实现，不需要先沉淀文档。
- 需求信息极少，连页面范围、业务目标和实现边界都无法判断。

## 必读参考

### 什么时候读骨架

一旦确定进入本技能，先读取 `references/前端技术文档骨架.md`，优先按这份骨架直接组织文档。

### 如何使用参考文件

- 优先使用骨架文件生成内容。
- 如果用户有团队既有格式要求，在当前输出中直接按要求调整，不再回退到旧模板示例文件。

## 输入要求

开始写文档前，优先确认是否已具备这些信息：

1. 业务目标：这个需求要解决什么问题。
2. 页面范围：涉及哪些页面、弹窗、流程节点或终端。
3. 交互流程：关键路径、状态变化、条件分支。
4. 数据来源：接口、缓存、本地状态、外部依赖。
5. 实现约束：技术栈、兼容性、性能、安全、上线边界。

如果缺少其中任一关键项：

- 先列出缺口。
- 能从当前上下文推导的，明确写成“假设”或“待确认”。
- 不能可靠推导的，不要编造。

## 输出结构

默认输出一份中文技术方案文档，至少覆盖以下结构：

1. 需求概述：背景、目标、页面范围、业务任务。
2. 技术实现：目录结构、任务拆分、组件拆分、公共功能、接口设计。
3. 非功能性需求：性能、兼容性、安全性、体验。
4. 风险评估与应对：明确风险来源、影响和缓解方案。

如果当前任务信息不足以写满所有章节：

- 保留章节。
- 用“待确认”“需补充”写明缺口。
- 不要用空标题占位后直接结束。

## 必须覆盖项

### 1. 需求概述

- 背景和业务目标。
- 页面或模块范围。
- 不在本次范围内的内容，如能判断则显式写出。

### 2. 页面与流程拆分

- 关键页面、弹窗、步骤页或终端差异。
- 主流程和条件分支。
- 用户操作如何驱动状态变化。

### 3. 技术实现

- 目录结构或模块落点。
- 任务拆分，至少拆到可分配给开发的粒度。
- 组件职责、输入输出、通信方式。
- 公共能力复用点，如 hooks、utils、store、event bus、缓存。

### 4. 数据与接口

- 每类数据从哪里来、如何转换、由谁持有。
- 接口用途、请求时机、错误处理、兜底策略。
- 如果接口未定，明确标记为待后端确认，不要自造字段。

### 5. 非功能性需求

- 性能：首屏、渲染、请求次数、长列表、缓存策略。
- 兼容性：浏览器、设备、容器、小程序或 PC 差异。
- 安全性：鉴权、越权、敏感信息暴露、前端校验边界。
- 用户体验：加载态、空态、错误态、回退路径。

### 6. 风险与应对

- 风险来源必须具体，例如接口不稳定、老状态机耦合、跨端差异、埋点缺失。
- 应对措施必须能执行，不能只写“加强沟通”“关注风险”。

## 写作流程

1. 先抽取需求事实，不要直接套模板。
2. 先读取骨架，必要时再读取模板，对齐本次文档的章节结构。
3. 把事实映射到章节，不够的信息标记为假设或待确认。
4. 优先写页面、流程、组件、数据流和接口关系。
5. 最后补非功能性需求和风险，不要只写功能点。
6. 输出前检查是否存在模板残留、示例脏数据和空洞章节。

## 产出要求

- 用中文写作，术语保持前端团队可读。
- 标题层级清晰，章节命名贴近前端研发语境。
- 内容以“可评审、可开发、可联调”为标准，不写泛泛描述。
- 允许保留少量待确认项，但必须显式标注其影响范围。

## 红线

### 绝对禁止

1. 把模板示例需求、示例 owner、示例章节原样带入新文档。
2. 只有功能点列表，没有页面拆分、组件职责、数据流或接口说明。
3. 用“后续实现时再看”“按常规处理”替代关键技术决策。
4. 在信息不足时编造接口字段、状态流转或组件边界。
5. 只写 happy path，不写异常、空态、失败态和风险。
6. 把这个技能当成中文排版润色器，而不产出前端技术方案。

### 借口与现实

| 借口 | 现实 |
|---|---|
| 模板已经很完整，直接改几个字就行 | 错。模板只提供骨架，不提供当前需求的事实。 |
| 先把功能点列出来，技术实现以后补 | 错。这个技能的目标就是输出技术实现文档。 |
| 接口大概都差不多，我先补几个字段 | 错。未确认的接口只能标为待确认，不能伪造。 |
| 风险先不写，开发时自然会知道 | 错。评审文档不写风险，等于没有方案闭环。 |
| 只要中文通顺就算完成 | 错。通顺不是目标，可开发性才是目标。 |

## 技能集成

`cdoc` 不作为 superpowers 总入口的独立路由目标，而是 `writing-plans` 输出后的后续可选步骤——当计划涉及前端交付时，`writing-plans` 会提示用户是否需要生成前端技术文档。

### 前置技能

- `superpowers:brainstorming`：需求边界不清、页面范围未定时，先收敛方案。
- `superpowers:writing-plans`：`cdoc` 依赖已有实现计划作为输入；没有计划时，先走 `writing-plans`。

### 后置技能

- `superpowers:verification-before-completion`：在宣称文档完成前，重新核对是否覆盖关键章节和待确认项。
